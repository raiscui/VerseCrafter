# 关键洞察: `pytorch3d` 构建错误引用不存在的 `nvcc`

## [2026-03-31 12:39:27 UTC] [Session ID: 019d43e0-6da5-7a33-910f-1a81ae037763] 主题: 显式工具链路径一旦允许覆盖, 就必须先做真实性校验

### 发现来源
- 本轮 `pytorch3d` 构建失败排查.
- 用户报错最终落在不存在的 `/usr/local/cuda-12.9/bin/nvcc`.

### 核心问题
- 对安装脚本来说, “允许用户显式覆盖 `CUDA_HOME`”本身没问题.
- 真正危险的是:
  - 脚本接收了覆盖值
  - 却不在入口验证这个目录里到底有没有 `bin/nvcc`
  - 最后把坏路径一路带到更深层的构建系统

### 为什么重要
- 深层构建失败的错误信息往往又长又晚.
- 用户最后只会看到一个远处的 `ENOENT`, 很难第一时间意识到根因其实是“入口参数就是错的”.
- 这会把排查时间浪费在 PyTorch3D、PyTorch、ninja 甚至编译器兼容性上.

### 当前结论
- 对这类“可显式覆盖的工具链路径”, 更稳的顺序应该是:
  - 先接受配置
  - 立刻验证路径真实性
  - 再进入 clone / build / pip install
- 文档示例也不该硬编码某个版本目录, 而应该引导用户从真实 `nvcc` 反推 toolkit 根目录.

### 后续讨论入口
- 后续如果继续整理 `bootstrap` 子任务, 可以优先审计:
  - `install-grounded-sam2`
  - `install-grounding-dino`
  - 其它接受显式 CUDA / compiler 路径的脚本

### 发现来源
- 本轮按用户要求在现有 Pixi 环境里覆盖安装 Blackwell 可工作组合.
- 安装后验证已通过, 但 `pixi.toml` 仍停留在旧的 `torch 2.3.1 / cu121` 约束.

### 核心问题
- 覆盖式 `pip install` 能快速救现场.
- 但只要声明式依赖文件没有同步, 下一次:
  - 重建环境
  - `pixi install`
  - 或其他机器重放
  就可能把这次已经验证成功的组合悄悄冲掉.

### 为什么重要
- 这类问题最容易制造“我这里明明好好的, 你那边怎么又坏了”的假象.
- 用户会误以为问题已经被永久修好, 实际上修复还停留在工作区现场状态.

### 未来风险
- Blackwell 机器重新 bootstrap 时回退到旧 PyTorch 组合.
- 团队成员或 CI 因复用了旧 `pixi.toml` 而重现安装失败或兼容性问题.

### 当前结论
- 覆盖式安装适合救火.
- 但只要项目打算长期支持 Blackwell, 就应该补一次“把已验证组合写回声明式环境”的收口动作.

### 后续讨论入口
- 若后续要做正式收口, 优先看:
  - `pixi.toml`
  - `README.md`
  - `scripts/install_pytorch3d.sh`
  - 本支线 `notes__pytorch3d_nvcc_path.md` 里的现场成功组合

## [2026-03-31 15:10:45 UTC] [Session ID: 019d43e0-6da5-7a33-910f-1a81ae037763] 主题: 新 CUDA 组合的“声明式真相源”未必在 conda, 有时必须诚实地写回 PyPI wheel 来源

### 发现来源
- 本轮尝试把 `torch 2.7.0+cu128` 写回 `pixi.toml`.
- 第一版按习惯继续使用 conda `pytorch + pytorch-cuda 12.8.*`, 结果被 `pixi install` 直接否定.

### 核心问题
- 现场成功运行的来源是:
  - PyTorch 官方 `cu128` wheel
- 但如果 manifest 仍执着于:
  - conda `pytorch-cuda`
  就会出现“你想描述的来源”和“你实际跑通的来源”根本不是同一个东西.

### 为什么重要
- 这类偏差不会只带来一个版本号差异.
- 它会让 lockfile、重建环境、CI 和团队复现全部偏离已经验证成功的现场.
- 最后出现的现象往往是:
  - 现场可用
  - 重装失败
  - 团队成员机器不一致

### 当前结论
- 声明式配置的核心不是“尽量统一走某一个包管理器”.
- 而是“忠实表达当前已被验证的真正来源”.
- 对本轮来说, 更诚实也更稳定的真相源是:
  - Pixi 管基础 conda 工具链
  - `torch / torchvision / torchaudio` 用 `pypi-dependencies + per-package index` 明确锁到 PyTorch `cu128` wheel

### 后续讨论入口
- 以后再遇到类似“新 CUDA / 新 GPU / conda 候选滞后”的场景时, 先问:
  - 当前现场成功来源到底是 conda 包, 还是上游 wheel?
  - manifest 写回去的是不是同一个来源?

## [2026-03-31 16:21:52 UTC] [Session ID: 479d2a22-a939-4b0b-a4a5-b682a4a82617] 主题: 第三方库的顶层便利导入会把“可选能力”偷偷变成“硬依赖”

### 发现来源
- 本轮 rendering_4D_control_maps.py 运行期失败排查.
- 表面需求只是一个深度反投影函数, 实际却被 kornia 顶层导入拖进 flash_attn.

### 核心问题
- 脚本作者以为自己依赖的是:
  - 一个小几何函数
- 但第三方库顶层 __init__ 实际帮它顺手导入了:
  - augmentation
  - contrib
  - feature
  - lightglue
  - flash_attn
- 于是“可选加速能力”在运行时被悄悄升级成了“模块导入阶段的硬门槛”.

### 为什么重要
- 这类问题的危险之处在于:
  - 堆栈会让人误以为根因在用户显式 import 的那一行
  - 实际坏的是第三方包内部更深的一条便利导入链
- 如果不把依赖边界收窄, 环境里任何一个可选加速库 ABI 漂移, 都可能把无关脚本一起拖死.

### 当前结论
- 对脚本级工具来说, 最稳的策略不是盲目相信第三方顶层 API “方便”.
- 更稳的是:
  - 先确认真实需求到底有多小
  - 如果只是基础几何或轻量逻辑, 优先本地化最小实现
  - 避免把整个脚本绑到一串并不需要的可选依赖上

### 后续讨论入口
- 后续如果继续审视其它推理脚本, 可以重点看:
  - 是否存在类似“只为了一个小函数就 import 整个大包”的情况
  - 是否有可替换为本地最小实现的基础几何 / 数据整理逻辑

## [2026-03-31 16:37:23 UTC] [Session ID: 2b367480-3f87-4242-8b24-ffdb2788e798] 主题: 对 Torch CUDA 扩展来说, “元数据存在”不是健康, import 成功才是健康

### 发现来源
- 本轮 flash_attn ABI 失配修复.
- 现场明明已经 import 失败, 但安装脚本仍因为 metadata.version("flash-attn") 存在而直接跳过.

### 核心问题
- Python 包管理层看到的是“发行版已安装”.
- 运行时真正需要的是“底层 .so 能被当前 torch ABI 成功加载”.
- 这两件事对纯 Python 包经常等价, 对 torch CUDA 扩展却经常完全不是一回事.

### 为什么重要
- 只看元数据会让 ABI 坏包在环境里长期潜伏.
- 用户会不断得到“已经安装”的假阳性反馈, 但一到真正 import 时还是炸.

### 当前结论
- 对 flash_attn、pytorch3d、Grounded-SAM-2 这类 torch CUDA 扩展, 安装入口的最小健康检查应该至少包含一次 import.
- 如果 import 失败, 就该把它视为“需要重装”, 不能继续沿用纯元数据短路.

### 后续讨论入口
- 后续可以审计 bootstrap 里其它 torch extension 安装脚本, 看是否也存在“只看 metadata 不看 import”的短路逻辑.
